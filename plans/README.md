## Dependency notes

- Plan 001 is self-contained.
- Plan 002 must not start until Plan 001 is DONE. It supplies app-owned
  runtimes and curated server payloads through Plan 001's bundled-command
  resolver.
- Plan 003 must not start until Plans 001 and 002 are DONE. It uses the fixed Pi
  MCP tools, code-mode `desktopMcp` API, desktop configuration approval, cached
  catalogs, Electron-owned stdio processes, and packaged server IDs without
  adding another bridge or machine runtime.

## Findings considered and rejected

- Tunneling raw MCP JSON-RPC through the device link would make the server own
  desktop process sessions, request IDs, reconnect behavior, and server-initiated
  requests. The existing device broker already provides authenticated,
  cancellable operations, so the plan forwards MCP operations instead.
- Registering every desktop MCP tool directly in Pi would make startup context
  and tool selection grow with every configured server. The plan exposes a
  fixed search, describe, call, and configuration surface.
- Windows On-device Agent Registry is not the baseline because Microsoft still
  marks it prerelease, it requires Windows build 26220.7262 or newer and package
  identity, and Sandi currently ships NSIS and portable builds. A future ODR
  source can replace the local config store inside the desktop host without
  changing the server protocol.
- Streamable HTTP, OAuth, MCP resources, prompts, sampling, elicitation, and
  experimental tasks are outside this plan. Chrome DevTools MCP and Windows-MCP
  both satisfy the requested desktop use case over stdio. Add another MCP
  transport or primitive only for a concrete server that needs it.
- AutoIt is not part of the baseline computer-use plan. Windows-MCP already
  provides semantic UI state, native input, window control, and wait primitives,
  while Plan 001 already gives Sandi a programmatic composition surface. Add a
  narrow app-specific automation only after a real application proves that its
  accessibility tree cannot support the requested workflow.
- Screenshot-driven clicking remains a fallback for canvases, remote desktops,
  and other visual-only controls. It is slower and more fragile than fresh
  semantic state, so it is not the default Windows interaction loop.
- Requiring users to install Node, Python, uv, npm, npx, or curated MCP packages
  is not part of the release design. Plan 002 ships locked Windows x64 runtimes
  and server payloads as application resources, so fresh NSIS and portable
  installs can start the supported servers without package downloads.
